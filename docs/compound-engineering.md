# Compound Engineering Pipeline

The task runner uses a compound pipeline — a multi-step agent workflow where each successful task triggers follow-up automation. Instead of a single "run and done" agent call, the runner chains stages to produce production-ready, reviewable output.

Each task runs in an **isolated git worktree** branched from the default branch, so the main checkout (where the bot and web UI run) is never disturbed.

## Pipeline stages

```
┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐
│ 0.Create │─► │ 1. Plan  │─► │ 2.Execute│─► │ 3.Docs   │─► │ 4.Commit │─► │ 5.Clean  │
│ Worktree │   │ Subtasks │   │ Task(s)  │   │ Update   │   │ & PR     │   │ Worktree │
└──────────┘   └──────────┘   └──────────┘   └──────────┘   └──────────┘   └──────────┘
```

### Stage 0 — Create worktree

An isolated git worktree is created at `/tmp/task-worktrees/task-<id>`, branched from `origin/<default-branch>`. If the task has a `target_repo` pointing to a local path, the worktree is created from that repo instead of `WORK_DIR`.

- Fetches latest from origin before branching
- Branch naming: `task/<id>-<slugified-title>`
- Cleans up stale worktrees from previous failed runs automatically
- Auto-installs dependencies: hard-links `node_modules` from the main checkout (fast, isolated); falls back to `pnpm install`

### Stage 1 — Plan checklist

A fast model decomposes the task into 2-5 concrete steps.

- Only runs when `AUTO_PLAN=true` (default) and the task is a top-level task with no existing subtasks
- Uses `MODEL_PLAN` for the planning pass — in code this is `MODEL_FULL` if set, otherwise `MODEL_FAST` (see `src/pipeline.py`). Configure models via `.env`, not hardcoded aliases
- Output is a JSON array of `{title, description}` objects
- If the plan returns only 1 step, the task runs directly (avoids overhead)
- Steps are created as child task records with `parent_id` set (for UI visibility)
- Planning results are appended to the parent task under `## Plan`
- If planning fails or returns no result, the task runs directly (single-step fallback)

### Stage 2 — Execute (single session)

The plan steps are assembled into a single compound prompt (checklist), and ONE agent call works through all steps in sequence. This gives the agent continuous context across steps.

- Runs via: `agent -p --force --model <model> <checklist-prompt>` with `cwd` set to the worktree
- If the task has a `role` set, the prompt is prefixed with `"You are acting as a <label>. <role prompt>"` — giving the agent a persona for the task (e.g. "Product Designer", "Frontend Engineer"). No prefix is added when `role` is empty.
- All child task records are marked `in_progress` before the call and `completed` after
- If the agent fails, all child tasks and the parent are set back to `pending`
- For simple tasks (no plan or 1-step plan), the original task prompt is used directly

### Stage 3 — Update docs

A second agent call reviews the changes made in stage 2 and updates any relevant documentation.

- Only runs when `AUTO_DOCS=true` (default) and a worktree was created
- Always uses `MODEL_FAST` for efficiency
- Output is appended under `## Doc Update`
- Failures are logged but don't block the pipeline

### Stage 4 — Commit & create PR

All changes in the worktree are committed and pushed, then a pull request is opened via the GitHub CLI.

- Only runs when `AUTO_PR=true` (default) and a worktree was created
- Commits all changes with message: `task(<id>): <title>`
- Pushes the task branch and opens a PR against the default branch
- PR body is generated by a fast model from the git diff (Summary, Changes, Testing sections)
- Falls back to diff stats if the summary generation fails
- The PR URL is appended to the task record under `## PR Created`
- Requires `gh` CLI to be authenticated (`gh auth login`)

### Stage 5 — Cleanup

The worktree directory and local task branch are removed. This always runs, even if earlier stages failed or timed out.

## Configuration

All compound pipeline settings are in `.env`:

| Variable | Default | Description |
|----------|---------|-------------|
| `AUTO_PLAN` | `true` | Decompose tasks into subtasks via a planning pass |
| `AUTO_DOCS` | `true` | Run the doc-update agent step after task execution |
| `AUTO_PR` | `true` | Auto-create PRs for agent changes |
| `GH_BIN` | `gh` | Path to GitHub CLI binary |
| `WORKTREE_BASE` | `/tmp/task-worktrees` | Directory where task worktrees are created |
| `TASK_TIMEOUT` | `900` | Timeout in seconds for the main task execution |
| `PLAN_TIMEOUT` | `120` | Timeout in seconds for the planning pass (override in `.env` as needed) |
| `MODEL_FAST` | `auto` or your chosen model | Execution, docs, PR body, self-review, comment replies |
| `MODEL_DEFAULT` | _(empty)_ | Middle tier when a task selects a tier that maps here (empty = CLI default) |
| `MODEL_FULL` | _(empty)_ | Planning, plan-only decomposition, directive runs, PR review; also used as `MODEL_PLAN` when set (empty = falls back to `MODEL_FAST` for planning) |
| `PIPELINE_LOG` | `./pipeline.log` | Path to structured JSONL activity log |

Set `AUTO_PLAN=false`, `AUTO_DOCS=false`, or `AUTO_PR=false` to disable individual stages. The task execution stage always runs.

## Comment-triggered agent replies

Users can post comments on tasks and trigger the agent to respond. The agent is given the full task context (title, description, previous output, all comments) and takes action or answers based on the latest comment.

- Triggered via `POST /api/tasks/{task_id}/comment` (sets `reply_pending`), `POST /api/tasks/{task_id}/reply`, or the UI; the **EC2 poller** spawns `run_task.py --reply <task_id>` while `reply_pending` is true
- Agent response is added as a comment by author `agent`
- Works on tasks in any status (not just pending)

## Task record sections

After a full pipeline run, a parent task record may contain these appended sections:

```markdown
---

## Plan (2026-03-16T12:00:00+00:00)

Decomposed into 3 steps (single-session checklist):

- **Add validation logic** `a1b2c3d4`
- **Refactor service layer** `e5f6g7h8`
- **Update tests** `i9j0k1l2`

---

## Agent Output (2026-03-16T12:00:30+00:00)

<output from the single agent session working through all steps>

---

## Doc Update (2026-03-16T12:01:00+00:00)

<output from doc update agent>

---

## PR Created (2026-03-16T12:01:30+00:00)

https://github.com/user/repo/pull/42
```

Child task records exist for UI visibility but don't have their own agent output.

## Observability

Every pipeline stage emits structured events to `pipeline.log` (JSONL format). Each event includes:

- `ts` — ISO timestamp
- `task_id` — which task
- `event` — what happened (`task_start`, `planning_done`, `subtask_done`, `pr_done`, etc.)
- `stage` — pipeline stage (`pipeline`, `plan`, `execute`, `docs`, `pr`, `worktree`, `cleanup`)
- `message` — human-readable description
- `extra` — optional fields: `runtime` (seconds), `model`, `parent_id`, `priority`

The web UI exposes this at `/activity` with filtering by task ID. Each task detail page includes total runtime and a link to its filtered activity log.

Discord notifications are sent only for top-level task status changes (subtasks are excluded).

## Design rationale

- **Separation of concerns**: The task prompt focuses on the actual work. Documentation is a separate concern handled by a dedicated prompt, so the task author doesn't need to remember to ask for doc updates.
- **Reviewability**: Auto-created PRs ensure that agent changes go through code review before landing on the default branch, rather than being committed directly.
- **Graceful degradation**: Each stage is independent. A doc-update timeout or a PR creation failure won't prevent the task from completing.
- **Opt-in/out**: All follow-up stages are togglable, so you can run the pipeline as a simple single-step runner by disabling them.
- **Single-session context**: By assembling plan steps into a single checklist prompt, the agent maintains context across all steps without redundant codebase scanning. This avoids the overlap and context loss that occurred with separate agent calls per subtask.
- **Conversational interaction**: Comment-triggered replies let users interact with the agent asynchronously — asking follow-up questions, providing additional context, or requesting further changes.

## Projects & daily directives

Projects are long-lived entities with a title, spec, priority, target repo, and status (`active`, `paused`, `completed`). They are stored in DynamoDB (`pk=PROJECT#<id>`, `sk=PROJECT`). Users provide daily directives — short instructions that guide one day's worth of agent work.

### Directive pipeline

When a directive is posted (`POST /api/projects/:id/directive`), the Hono API writes a `DIR#<iso-timestamp>` record and triggers `run_task.py --directive <project_id> <directive_sk>` on EC2 via SSM. `run_directive()` in `pipeline.py`:

1. Fetches the project spec and directive content
2. Runs a fast agent (plan-only style) to decompose into independent tasks
3. Creates tasks with `project_id`, `directive_sk`, and `directive_date`
4. Sets inter-task dependencies
5. Triggers runners for unblocked tasks

Each task carries project context — `build_prompt()` prepends the project spec so the agent has full awareness of the product/technical spec.

### Daily boundary

When all tasks in a directive batch reach terminal status (`completed`, `in_review`, `cancelled`), the `maybe_finalize_directive_batch()` method on the store sets `awaiting_next_directive: true` on the project. `pick_next_task()` skips tasks belonging to a finalized batch or paused projects. The human reviews the day's output and posts a new directive to continue.

### Daily digest Lambda

A scheduled Lambda (`infra/packages/digest/`, cron `0 14 * * ? *`) scans DynamoDB for the last 24h of activity, aggregates task counts by status and project, checks EC2 health, and posts a summary to Discord via webhook. This gives the human a daily rollup without needing to open the UI.
